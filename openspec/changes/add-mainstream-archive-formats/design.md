## Context

当前 `ArchiveReader` 只有 `ZipArchiveReader` 和 `PlainReader`。`open_archive` 通过 ZIP 扩展名或 `PK` 文件头分派，其余文件全部进入裸文本读取器。因此 `.7z` 等二进制文件会在用户点击后报“不是文本日志”，目录监控也只把 ZIP 作为归档候选。

新增格式同时影响格式检测、文件到达通知、目录树惰性展开、条目流、索引缓存、错误翻译和跨平台打包。ZIP 可以读取中央目录并对 Stored 条目随机访问；TAR、7z、RAR 与压缩流通常需要顺序扫描，不能假设所有格式都具有 ZIP 的元数据和 seek 能力。

## Goals / Non-Goals

- Goals:
  - Windows 与 macOS 开箱即用地读取日志场景中的主流归档。
  - 所有模块使用同一格式检测结果，不出现“能通知但不能打开”或“能打开但不通知”。
  - 不向用户目录解压，不整条目入内存，失败与取消后无残留缓存。
  - 自动识别并逐层打开受支持的嵌套归档，保持路径、进度和缓存生命周期可追踪。
  - 保留 ZIP Stored 条目的有界 seek 优化，其它格式按真实能力暴露顺序流。
  - 对依赖许可证、原生库、包体积和安全更新建立发布门槛。
- Non-Goals:
  - 不在本 change 支持密码输入或分卷归档。
  - 不创建、修改、追加、删除条目或重新打包归档。
  - 不承诺所有非标准压缩算法、损坏容错模式或自解压可执行文件。

## Supported Matrix

| 类型 | 扩展名 | 首版行为 | 访问特性 |
|---|---|---|---|
| ZIP | `.zip` | 保留现有支持 | 中央目录；Stored 可 seek，Deflate 顺序流 |
| 7z | `.7z` | 列出并打开未加密单卷条目 | 通常顺序解码 |
| RAR | `.rar` | 列出并打开未加密单卷 RAR4/RAR5 条目 | 顺序解码；受依赖许可验收约束 |
| TAR | `.tar` | 列出并打开普通文件条目 | 顺序扫描 |
| gzip TAR | `.tar.gz`, `.tgz` | 解压外层并读取 TAR | 顺序扫描 |
| bzip2 TAR | `.tar.bz2`, `.tbz`, `.tbz2` | 解压外层并读取 TAR | 顺序扫描 |
| xz TAR | `.tar.xz`, `.txz` | 解压外层并读取 TAR | 顺序扫描 |
| zstd TAR | `.tar.zst`, `.tzst` | 解压外层并读取 TAR | 顺序扫描 |
| 单文件压缩流 | `.gz`, `.bz2`, `.xz`, `.zst` | 合成一个日志条目，名称优先使用流内原名，否则移除最后一层后缀 | 顺序解码 |

WIM、`.zipx`、CAB、ISO、DMG 和自解压 EXE 不属于本次“日志归档格式”范围。WIM 更接近包含一个或多个 Windows 系统映像的部署容器，读取它通常需要引入 `wimlib` 等原生依赖，因此当前版本明确不支持，也不会将 `.wim` 宣称为可读取归档。

支持矩阵中的任意归档容器 MAY 作为另一受支持归档的普通文件条目出现，例如 `outer.zip → diagnostics.tar.gz → service.7z → app.log`。嵌套层级使用同一格式注册表识别，不维护单独的组合白名单。

## Decisions

### 单一格式注册表

新增 `ArchiveFormat` 与格式注册表，集中保存：规范名称、复合后缀、magic 检测、是否单文件流、读取器工厂和 UI 类型标识。watcher、拖入、目录树与 `open_archive` 只消费注册表结果，不再各自硬编码 `.zip`。

检测优先级：

1. 已知 magic（ZIP、7z、RAR、gzip、bzip2、xz、zstd）；
2. 对压缩流解码小型前缀，判断内层是否为 TAR；
3. 未压缩 TAR 使用头部校验与复合后缀辅助；
4. magic 与后缀冲突时以可验证内容为准，并在错误中保留诊断信息。

### 读取器分层

- 保留专用 ZIP 读取器，避免丢失中央目录和 Stored seek 优势。
- TAR 与单文件压缩流优先使用小型、可审计的 Rust 解码器组合。
- 7z/RAR 在实施前比较“格式专用 Rust crate”与“统一 libarchive 后端”。选择条件依次为：Windows/macOS 一致性、流式 API、安全维护状态、许可证与再分发、包体积。
- 若使用原生 libarchive，必须锁定版本、随安装包提供所需动态库、生成依赖清单，并验证 updater 安装后仍能加载；不得依赖用户预装命令行工具。

### 惰性扫描与缓存

ZIP 展开仍只读中央目录。顺序归档只在用户首次展开时扫描条目元数据，不在监控阶段打开。相同文件版本的条目清单可以使用有界缓存；文件大小或修改时间变化、节点删除、会话关闭或 LRU 回收时失效。

打开条目时重新定位并返回顺序 `Read` 流。现有索引层把已读取字节写入应用缓存，以支持大日志滚动；归档读取器本身不释放用户可见文件。缓存路径必须继续使用应用缓存目录，并在关闭、取消、失败和启动清理时移除。

### 嵌套归档递归模型

归档条目名称或格式元数据表明其可能是受支持归档时，目录树将其呈现为可展开候选；用户展开该节点时再读取有限前缀验证 magic，并调用相同格式注册表创建下一层读取器。系统不会在展开外层时自动扫描所有后代，避免嵌套炸弹和无界 IO。

嵌套路径使用稳定的条目链表达，例如 `outer.zip::diagnostics.tar.gz::service.7z::app.log`。每层都保留父归档文件身份与条目身份，防止同名嵌套条目混淆，并支持通知定位、选项卡标题和错误上下文。

若下一层读取器需要 `Seek` 而父层只能提供顺序流，系统 MAY 将该嵌套归档条目流式写入应用缓存目录，再从缓存建立读取器。中间缓存不得写入用户目录、不得整文件进入内存，并必须计入聚合实际解码字节上限。

默认最大嵌套深度为 5 层；单次嵌套链的累计实际解码字节不得超过现有单条目安全上限。达到深度或字节上限时停止展开并显示明确错误。折叠节点 MAY 保留有界缓存以便再次展开；关闭关联选项卡、父归档变化、LRU 回收、失败或应用启动清理时必须删除。

### 安全边界

- 继续以实际解压字节数执行单条目上限，不信任归档声明大小。
- 嵌套归档同时执行最大 5 层深度与整条链累计实际解码字节上限。
- 增加归档条目数量、路径长度、元数据扫描字节数和扫描时长上限。
- 只读取普通文件条目；忽略目录、符号链接、硬链接、设备节点和其它特殊条目。
- 不把归档路径用于磁盘写入；规范化显示路径并拒绝 NUL、绝对路径和危险父级片段作为可打开日志条目。
- 损坏校验和、截断流、未知算法、加密与分卷必须快速失败并可取消。

### RAR 发布门槛

RAR 功能属于要求范围，但在选定依赖前必须记录：许可证文本、商业/非商业使用条件、二进制再分发权、RAR4/RAR5 覆盖、已知 CVE 更新机制和 Windows/macOS 包体积。任何一项不明确时不得发布“支持 RAR”的正式构建，界面仍应显示明确的暂不支持原因。

## Risks / Trade-offs

- 顺序格式展开较慢 → 仅按需扫描、支持取消、显示“正在读取归档目录”，并缓存同一文件版本的清单。
- 原生依赖增大安装包或引入加载失败 → 优先纯 Rust；CI 对安装包大小和运行时动态库做双平台检查。
- RAR 许可或实现质量不足 → 以发布门槛阻止虚假支持，不使用用户机器上不确定的外部命令。
- 格式数量增加扩大攻击面 → 夹具模糊测试、资源上限、固定版本和依赖审计纳入 CI。
- 单文件 `.gz` 与 `.tar.gz` 判定歧义 → 解码有限前缀识别 TAR，而不是仅依赖文件名。
- 嵌套归档可放大 CPU、IO 与磁盘缓存 → 逐层惰性展开、深度 5、聚合字节上限、取消和生命周期清理共同约束。

## Migration Plan

1. 建立格式注册表并让现有 ZIP/裸文本通过注册表，行为保持不变。
2. 实现 TAR 和 gzip 单文件/TAR，验证顺序读取与缓存边界。
3. 增加 bzip2、xz、zstd 组合格式。
4. 完成 7z 后端与安全测试。
5. 完成 RAR 依赖许可评审、读取器和双平台打包。
6. 实现嵌套归档条目链、逐层格式识别、中间缓存与深度/聚合字节限制。
7. 统一 watcher、拖入、通知、图标、错误文案和 README 支持矩阵。
8. 用每种格式及跨格式嵌套组合的真实夹具执行 Windows/macOS 验收。

回滚时按格式注册项关闭新增读取器；ZIP 与裸文本路径保持可用，配置格式不需要迁移。
