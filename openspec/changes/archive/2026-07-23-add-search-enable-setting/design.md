## Context

搜索后端目前在应用后台状态初始化完成后立即调用 `resume_or_watch`。前端设置在 WebView 加载后才可用，因此仅使用 `localStorage` 无法阻止启动阶段的搜索恢复。与此同时，运行中销毁索引 watcher、provider 和服务连接会引入额外竞态，不符合“下次启动生效”的要求。

## Goals / Non-Goals

- Goals:
  - 默认不启动全局搜索后台工作。
  - 将当前进程的有效状态与下一次启动偏好明确分离。
  - 所有搜索入口和后端命令使用同一个有效状态。
  - 保留已有本地索引，以便重新启用时快速恢复。
- Non-Goals:
  - 运行中动态启动或停止搜索 provider。
  - 关闭搜索时自动清除索引。
  - 改变搜索算法、搜索结果或索引服务安装状态。

## Decisions

- Decision: Rust 后端在应用配置目录保存版本化搜索功能偏好，默认 `enabled = false`。应用初始化时读取一次并冻结为当前进程有效状态，然后决定是否调用 `resume_or_watch`。
- Decision: 设置命令只原子保存下一次启动偏好并返回当前进程有效状态与已保存偏好，不改变当前 `FileSearchManager` 生命周期。
- Decision: 前端设置开关展示已保存偏好；当它与当前进程有效状态不一致时展示待重启状态。每次实际切换都显示下次启动生效提示。
- Decision: 搜索入口以当前进程有效状态为准。未启用时顶栏入口显示为禁用，悬停提示用户在设置中开启搜索本地文件功能，快捷键不执行操作；后端命令仍执行独立检查，避免前端状态过期绕过限制。
- Decision: 搜索已启用时，顶栏搜索入口是监控目录页与搜索页之间的切换开关；搜索页中保持激活样式并通过悬停文案说明再次触发将返回监控目录。搜索快捷键使用同一切换逻辑。
- Decision: 禁用搜索不删除 SQLite、Tantivy 或 provider 快照。用户可通过现有“清除索引”操作显式删除数据。

## Risks / Trade-offs

- 用户开启后不能立即搜索 → 在开关旁和每次切换后明确提示需要重启。
- 用户关闭后当前进程仍有索引活动 → 设置显示“下次启动后关闭”，避免把偏好误报为当前状态。
- 配置损坏或写入失败 → 损坏时安全回退关闭；写入失败时保持原值并向用户显示错误。

## Migration Plan

1. 缺少偏好的现有安装按默认关闭处理，但保留已有搜索索引。
2. 用户在设置中开启并重启后恢复既有索引或按原流程建立索引。
3. 用户关闭并重启后不恢复搜索后台任务；再次开启不会丢失索引。
