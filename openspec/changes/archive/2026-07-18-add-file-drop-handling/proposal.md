# Change: 新增拖入文件监控与日志打开

## Why

用户已经在文件管理器中定位到文件或目录时，仍需先手动添加监控根、再在 LogPeek 目录树中逐层查找。将任意文件或文件夹直接拖入应用可以统一完成监控配置；其中只有可识别的裸文本日志还应立即打开查看。

该交互必须明确“加入监控”的真实含义并保证路径安全。本 change 将其定义为：文件使用其父目录、文件夹使用自身作为监控根，不移动、复制、重命名或删除原路径。

## What Changes

- 监听 Tauri 主 WebView 的原生文件拖放事件；拖入悬停和处理中不显示遮罩、弹窗或其它提示层。
- 本 change 一次仅接受一个本地普通文件或文件夹；多路径拖入整体拒绝并留给后续独立 change。
- 后端规范化并检查拖入路径，返回文件、归档或目录类型、实际监控路径、文本日志判定和是否已在监控范围，前端不自行信任扩展名。
- 拖入任意普通文件时直接将其父目录加入监控；拖入文件夹时直接将文件夹自身加入监控，不再显示加入监控确认弹窗。
- 拖入 `.log`、`.txt`、`.json` 等可识别裸文本日志时立即打开正文；无论用户是否同意监控目录，直接查看都继续执行；监控已存在或添加成功时还会在目录树中逐层定位该日志。
- 规范化后已被现有监控根覆盖的路径不重复询问或添加；新增父级监控根时移除被其覆盖的子级监控根，确保配置与 watcher 均不存在父子重复监控。
- 增加路径检查、任意文件和文件夹分流、日志自动打开、自动监控时序、重叠监控根归并、多路径拒绝和跨平台拖放测试。

## Impact

- Affected specs: `file-drop-handling`（新增）
- Affected code: `src/App.tsx`、`src/api/tauri.ts`、`src-tauri/src/lib.rs`、文件分类与监控范围检查
- Dependencies: 复用现有 `@tauri-apps/api` 的 `onDragDropEvent`，不新增 npm 或 Cargo 依赖
- Data: 用户完成单路径拖放后自动复用现有监控目录配置持久化；不新增配置格式
- Platforms: Windows 与 macOS 使用 Tauri 原生绝对路径事件；浏览器 mock 模式不尝试获取真实文件路径
