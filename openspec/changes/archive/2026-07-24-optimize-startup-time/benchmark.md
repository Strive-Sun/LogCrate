# Windows 启动性能记录

## 测试环境

- 日期：2026-07-24
- 系统：Windows 10 专业版 10.0.19045
- CPU：Intel Core i7-10700K 3.80 GHz
- 内存：15.8 GB
- 构建：`cargo build --release --bin logcrate --features custom-protocol`
- 搜索配置：已开启，根目录为 `C:\`、`D:\`
- 测量点：进程入口到 React 主页面加载层移除后的首个绘制帧

## 优化前阶段基线

接入阶段计时但尚未加入可交互门控时运行 3 次：

| 样本 | 可交互耗时 |
| --- | ---: |
| 1 | 1469.8 ms |
| 2 | 1608.0 ms |
| 3 | 1739.6 ms |

P50 为 1608.0 ms，最大值为 1739.6 ms。后台 watcher 与搜索在约 371 ms 时已经开始调度，早于约 1.6 秒的可交互帧，存在与 WebView 冷启动争用 CPU、磁盘和数据库锁的风险。

## 开发构建补充验证

启动 Vite 后对 `target/debug/logcrate.exe` 连续测量 3 次，结果为 2652.2 ms、1617.4 ms、1500.2 ms，全部低于 3 秒。首个样本包含开发服务器首次模块转换，其余样本稳定在约 1.5–1.6 秒。该结果仅用于确认 `tauri:dev` 不再复现二十秒等待，不替代正式的 Release 验收。

用户环境随后捕获到一次 24661.2 ms 的 `tauri:dev` 启动，其中 WebView 在 741.0 ms 创建完成、1647.2 ms 开始导航，但直到 24435.5 ms 才完成页面加载。进一步确认主要问题集中在开发 URL 的本机解析/代理及资源加载路径，同时屏幕外 1×1 子 WebView 也存在被 Windows WebView2 判定为不可见并节流的风险。

修复后将 Vite 与 Tauri `devUrl` 固定到 `127.0.0.1`，并在 Windows 上从创建开始使用全尺寸透明 WebView，使底层原生进度画面保持可见而 WebView 不再处于离屏状态。真实 `npm run tauri:dev` 复测结果：

| 阶段 | 修复前 | 修复后 |
| --- | ---: | ---: |
| WebView 创建完成 | 741.0 ms | 964.6 ms |
| 导航开始 | 1647.2 ms | 1057.9 ms |
| 导航完成 | 24435.5 ms | 2385.9 ms |
| 主页面可交互 | 24661.2 ms | 2611.6 ms |

开发启动从 24.66 秒降低到 2.61 秒，后台 watcher、搜索和缓存维护仍在 2.61 秒可交互之后才开始。

用户实机再次捕获到 26600.7 ms 的启动，其中 `navigation-started` 到 `navigation-finished` 独占约 23.97 秒。运行中的 Vite 服务实测首页、`main.tsx`、`App.tsx` 和样式入口响应均为 1–6 ms，同一开发 URL 由 Edge 完整加载约 2.1 秒；但这些测量发生在模块已经完成转换和缓存之后，只能排除稳态资源响应慢，不能排除首次导航时的 Vite 冷转换链。后续逐请求计时证明，早期把问题主要归因于 WebView2 本地导航环境并不充分。

Windows WebView 随后显式禁用系统代理、后台网络和组件更新，避免代理软件或 WebView2 组件任务干扰 `127.0.0.1` 开发导航。重新创建进程后的真实 `npm run tauri:dev` 结果：

| 阶段 | 耗时 |
| --- | ---: |
| 原生首帧 | 50.9 ms |
| WebView 创建完成 | 970.1 ms |
| 导航开始 | 1000.5 ms |
| 导航完成 | 2356.6 ms |
| React 挂载 | 2402.5 ms |
| 主页面可交互 | 2549.2 ms |

此次复测在 2.55 秒达到可交互状态；watcher、搜索和托盘仍在可交互握手后才开始调度。但用户随后仍能稳定复现二十多秒等待，说明代理和 WebView2 后台请求优化只能减少波动，不是充分的根因修复。

用户实机继续复现约 27.40 秒后，为 Vite 增加临时逐请求计时并在同一环境成功复现。冷启动时 `/index.html` 响应耗时 3686.5 ms、`/@vite/client` 耗时 1904.9 ms、`/src/main.tsx` 耗时 10079.6 ms，后续依赖模块直到约 28 秒才开始集中返回。该结果最终确认瓶颈是 Vite 开发模式的按需转换与多模块请求链，而不是 WebView2 页面执行、Rust 初始化或业务后台任务。

默认 `npm run tauri:dev` 因此改为先把前端构建到独立的 `.tauri-dev-dist`，再由 Vite preview 静态服务加载单 bundle；该目录与正式 `dist` 隔离，避免每次开发启动触发 Rust 资源重新嵌入和链接。需要前端 HMR 时可显式运行 `npm run tauri:dev:hmr`。静态开发路径实测：

| 阶段 | 耗时 |
| --- | ---: |
| 原生首帧 | 77.5 ms |
| WebView 创建完成 | 780.0 ms |
| 导航开始 | 798.0 ms |
| 导航完成 | 847.5 ms |
| React 挂载 | 908.3 ms |
| 主页面可交互 | 1072.6 ms |

页面导航仅耗时 49.5 ms，主页面在 1.07 秒达到可交互状态。首次执行前的前端构建约 1 秒；该构建发生在应用进程启动之前，不会延长原生进度页面到主页面的交接。

最终以未修改任何文件的日常增量场景再次执行默认命令：前端静态 bundle 构建 1.02 秒、Cargo 增量检查 1.11 秒、应用进程到主页面可交互 874.2 ms，命令执行到主页面合计约 3 秒；其中页面导航仅耗时 45.0 ms。
