## Context

应用当前由 Tauri 默认事件循环管理，未创建托盘图标，也未拦截主窗口 `CloseRequested`。直接让前端监听关闭事件会把关键生命周期逻辑放进 WebView，并可能在页面尚未加载、崩溃或更新重启时失效，因此关闭决策和托盘操作应由 Rust/Tauri 层统一负责。

## Goals / Non-Goals

- Goals:
  - 用户点击窗口关闭按钮后主窗口消失，但进程、目录监控和通知能力继续运行。
  - 托盘提供稳定的恢复入口和唯一的用户主动完整退出入口。
  - 自动更新重启和操作系统退出不被“关闭到托盘”逻辑阻断。
  - Windows 与 macOS 使用相同生命周期状态机。
- Non-Goals:
  - 本 change 不增加“关闭按钮行为”设置项或首次提示。
  - 本 change 不实现开机启动、单实例或后台服务。
  - 本 change 不改变窗口最小化按钮的原生行为。

## Decisions

### 在 Rust 层拦截窗口关闭

- 使用 Tauri 窗口事件处理主窗口 `CloseRequested`，调用 `prevent_close` 后隐藏窗口。
- 只拦截用户关闭窗口的事件，不拦截应用级 `exit`/`relaunch`，因此 updater 和系统退出路径仍可结束进程。
- 隐藏而非销毁 WebView，恢复时保留目录展开、当前会话和未读状态。

### 单一托盘与显式退出

- 应用启动时创建一个托盘图标和包含“显示主窗口”“退出 LogPeek”的菜单。
- 托盘左键释放（平台支持时）和“显示主窗口”执行同一恢复函数：取消最小化、显示并聚焦主窗口。
- “退出 LogPeek”直接调用应用级退出，不通过窗口关闭事件，避免再次被隐藏逻辑拦截。
- 托盘图标复用应用默认图标，不增加独立位图资源和安装包体积。

### 可测试的生命周期决策

- 将菜单动作和关闭动作归一为小型生命周期决策函数/枚举，纯逻辑覆盖“隐藏、显示、退出”。
- Tauri 集成部分通过编译检查和 Windows/macOS 人工验证确认：任务管理器进程存活、托盘恢复、托盘退出和 updater 重启。

## Risks / Trade-offs

- 用户可能误以为应用已退出 → 托盘图标和明确的“退出 LogPeek”菜单提供持续状态与退出入口。
- 托盘图标被系统折叠到隐藏区域 → 仍保持后台运行；用户可从系统托盘展开区找到 LogPeek。
- macOS 菜单栏交互与 Windows 右键习惯不同 → 使用平台原生托盘菜单事件，但保持相同菜单语义。
- 错误拦截所有退出会阻断更新 → 仅处理窗口 `CloseRequested`，应用级退出和 updater relaunch 不走该分支。

## Migration Plan

无需配置迁移。升级后关闭按钮行为立即变为隐藏到托盘；卸载、更新和操作系统关机流程保持原有行为。
