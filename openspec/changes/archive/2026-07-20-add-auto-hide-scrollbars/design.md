## Context

应用当前直接使用 WebView 原生滚动条。CSS 可以让滑块保持透明，但无法知道用户何时开始或停止滚动；若为每个目录树、日志选项卡和弹窗分别绑定监听，又会产生重复逻辑和清理风险。

## Goals / Non-Goals

- Goals:
  - 所有应用内主要滚动区域默认不显示滚动条。
  - 滚动发生时立即显示，停止后延迟并渐隐。
  - 同时滚动多个区域时，各区域的显示与计时互不影响。
  - 不影响日志虚拟滚动性能和现有布局宽度。
- Non-Goals:
  - 不实现自绘滚动条、滚动位置缩略图或滚动条尺寸配置。
  - 不隐藏操作系统窗口本身的边框或滚动控件。

## Decisions

### 使用单个捕获阶段滚动监听

- 在应用根组件挂载一个 `document` 级捕获监听。原生 `scroll` 事件不冒泡，但可在捕获阶段统一接收，避免为动态生成的折叠菜单和弹窗逐一注册。
- 事件目标发生滚动时添加 `is-scrolling` 类；每个元素维护独立的空闲计时器，后续事件只重置该元素计时。
- 空闲约 700ms 后移除类并开始约 450ms 的渐隐。
- 卸载时移除监听、取消全部计时器和残留类。

### 使用透明原生滚动条而非自绘覆盖层

- 保留 WebView 原生滚动机制和命中区域，仅控制轨道、滑块与角落的颜色；这不会改变 `scrollTop`、`scrollLeft` 或虚拟列表测量。
- Windows WebView2 与 macOS WKWebView 使用 `::-webkit-scrollbar` 系列样式；同时设置标准 `scrollbar-color` 作为兼容回退。
- 通过已注册的颜色自定义属性驱动滑块颜色变化：进入滚动态时取消过渡并立即显示，退出滚动态时执行颜色渐变。
- 滚动条宽度维持窄尺寸，透明时不产生可见轨道；因此正文不会因显示/隐藏而横向抖动。

### 可访问性与降级

- `prefers-reduced-motion: reduce` 下把过渡时间设为零，避免不必要动画。
- 若 WebView 不支持颜色自定义属性动画，则允许滚动条在空闲延迟后直接隐藏；滚动与内容访问能力不受影响。
- 仅对实际可滚动的目录树、日志正文、折叠菜单和更新说明区域启用样式，避免给普通元素增加无意义的样式计算。

## Risks / Trade-offs

- 透明状态下滚动条不易用鼠标直接发现 → 保留原生命中区域，滚轮、触控板和键盘行为不变，滚动一发生即显示。
- 高频日志滚动可能反复设置状态 → 每次事件仅执行一次 class 添加和单个计时器重置，不触发 React 渲染。
- 不同 WebView 对滚动条动画支持存在差异 → 使用标准颜色回退；最差情况为延迟后直接隐藏，不影响功能。

## Migration Plan

无需数据迁移。删除全局监听和自动隐藏样式即可回滚为持续显示的原生滚动条。
